    Dashboard
    
          
      
            
    • Shows live node_id, readiness, requests, uptime, beacons, and events.
      • 
            
    • Status: ready = LLM connected; stub = echo mode; degraded = Ollama unreachable.
      • 
            
    • Health check calls GET /health and shows raw JSON diagnostic data.
      • 
          
    
        
    
        
    
          
    Config
    
          
      
            
    • Set ollama_url (e.g. http://host.docker.internal:11434) and model name.
      • 
            
    • Test connection fires a ping inference and reports which backend responded.
      • 
            
    • Changes are applied live via PATCH /v1/config — no restart needed.
      • 
          
    
        
    
        
    
          
    Agents
    
          
      
            
    • Each agent has a name, optional persona, description, and system_prompt.
      • 
            
    • Clone duplicates an agent with a new ID — useful for A/B testing prompts.
      • 
            
    • Load Preset imports from /v1/agents/presets (defined in deploy/presets/agents.json).
      • 
            
    • Agents used in chains are referenced by their UUID.
      • 
          
    
        
    
        
    
          
    ATM (Automated Thought Machine)
    
          
      
            
    • Think: processes a prompt through the full reasoning pipeline.
      • 
            
    • Reflect: meta-analyzes a previous thought for consistency and gaps.
      • 
            
    • Use think-ahead suggestions to explore related follow-up prompts after a response.
      • 
            
    • The Thinking Path Planner breaks a high-level goal into a sequence of ATM steps.
      • 
          
    
        
    
        
    
          
    ORC (Observer Rights & Compliance)
    
          
      
            
    • Rights are read-only — they reflect the node's current ORC configuration.
      • 
            
    • Witness statements are logged into the DAG as compliance events.
      • 
          
    
        
    
        
    
          
    Mesh Peers
    
          
      
            
    • Peers are other Andyria nodes reachable over the network.
      • 
            
    • Adding a peer URL registers it for mesh communication via POST /v1/peers.
      • 
            
    • Green dot = reachable in last check; grey = unreachable or unknown.
      •

    Inference
    
          
      
            
    • POST /v1/infer — {input, session_id, context} → {output, model_used, processing_ms, confidence}

    Status & Health
    
          
      
            
    • GET /v1/status — full node status JSON
      • 
            
    • GET /health — lightweight health check
      • 
            
    • GET /metrics — Prometheus-format metrics
      • 
          
    
        
    
        
    
          
    Config
    
          
      
            
    • GET /v1/config — read current node config
      • 
            
    • PATCH /v1/config — {ollama_url?, ollama_model?}
      • 
          
    
        
    
        
    
          
    Agents
    
          
      
            
    • GET /v1/agents — list all agents
      • 
            
    • POST /v1/agents — create agent
      • 
            
    • PATCH /v1/agents/{id} — update fields
      • 
            
    • DELETE /v1/agents/{id} — soft delete (archive)
      • 
            
    • POST /v1/agents/{id}/clone — duplicate
      • 
            
    • GET /v1/agents/{id}/avatar.svg — generated avatar
      • 
            
    • GET /v1/agents/presets — built-in preset definitions
      • 
          
    
        
    
        
    
          
    ATM
    
          
      
            
    • POST /v1/atm/think — {input} → {output, reasoning_chain}
      • 
            
    • POST /v1/atm/reflect — {input} → reflection response
      • 
          
    
        
    
        
    
          
    Chains & Memory
    
          
      
            
    • GET|POST /v1/chains — list / create chains
      • 
            
    • POST /v1/chains/{id}/run — {input}
      • 
            
    • GET /v1/memory/{file} — read memory file
      • 
            
    • POST /v1/memory — {file, content} write
      • 
          
    
        
    
        
    
          
    Mesh & Sessions
    
          
      
            
    • GET|POST /v1/peers — list / add peers
      • 
            
    • GET /v1/session/{id} — inspect session
      • 
            
    • DELETE /v1/session/{id} — clear session
